Turning this into a real knowledge graph

The map above is hand-built from the repo — accurate today, stale the moment someone merges. Databricks already emits the graph as data; the work is reading it, not drawing it. Four steps, cheapest first.

01

Harvest the edges you already own

system.access.table_lineage records every read/write Databricks observes, with the notebook or job that did it. That is a producer→consumer edge list for free — no instrumentation, no parsing. column_lineage gives the same at column grain. notebooks/lineage/impact_analysis.sql already walks it recursively; it just isn't persisted anywhere. 

SELECT source_table_full_name  AS src,
       target_table_full_name  AS dst,
       entity_type, entity_id,          -- the job/notebook = the edge label
       max(event_time)         AS last_seen
FROM system.access.table_lineage
WHERE source_table_catalog = 'cdp_dev'
GROUP BY ALL
02

Add the edges lineage can't see

Runtime lineage only knows what ran. It misses the vector index → agent hop, the LLM calls, and anything gated behind a broken job. Those come from static sources: resources/*.yml for job→file and task→task, the DLT event log for flow-level dependencies, and the agent resources declared at mlflow.log_model time — which is exactly the index-and-endpoint list the graph above draws.

03

Land it as two tables, not a graph database

ops.kg_node and ops.kg_edge in Unity Catalog. Resist Neo4j here: the graph is a few hundred nodes, the queries are reachability over a shallow DAG, and a recursive CTE handles that fine. Keeping it in UC means it inherits governance, time travel, and the same SQL surface as everything else — and any agent can already query it.

Node key = the full three-part name. Edge = (src, dst, via_file, operation, last_seen). Refresh nightly off step 01, upsert step 02 on deploy.

04

Make drift a query, not a discovery

This is the payoff, and this repo makes the case on its own. Every red edge above is a name that exists in one file and nowhere in the catalog — gold.revenue_pipeline vs the deployed gold.gold_revenue_pipeline. With the graph as a table, that whole class of bug becomes an anti-join against information_schema.tables that runs in CI.

Same shape finds the dangling nodes: bronze tables with no outbound edge are ingest you're paying for and nobody reads. Both checks are a dozen lines once the edges are in a table.
